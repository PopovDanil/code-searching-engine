## Гибрид BM25 + dense (RRF) и переосмысление метрик ретривера (13.07.2026)

### Что и зачем добавили

1. **Гибрид BM25 + dense через RRF** (идея из CodeRAG). Реализация: `src/retrieval/bm25_index.py`
   (BM25Okapi + токенизация идентификаторов: `read_json_file`→`read json file`,
   `parseHTTPResponse`→`parse http response`), `src/retrieval/fusion.py` (Reciprocal Rank Fusion,
   `score=Σ w_i/(k+rank_i)`, k=60). Флаги конфига `enable_hybrid`, `hybrid_rrf_k`,
   `hybrid_dense_weight`, `hybrid_bm25_weight`.
2. **Метрики ретривера.** Раньше eval обрезал подсчёт ранга на 10 — нельзя было увидеть,
   попал ли ответ в пул top-50/100. Добавили **Recall@20/50/100, Precision@100, NDCG@100**;
   пул поднят до `retrieval_top_k=100`. Логика: у ретривера (первой ступени) задача —
   полнота пула кандидатов для реранкера (Recall@50/@100, NDCG@100), а точность на первом
   месте (Recall@1/MRR/NDCG@10) — это работа реранкера. Precision@100 при одном релевантном
   документе на запрос = Recall@100/100 (добавлен по просьбе, справочно, малоинформативен).

### История с гибридом (важно для рассказа)

- **Наивный гибрид (равные веса 1.0/1.0) УХУДШИЛ качество:** Qwen dense R@1 0.839 → hybrid 0.688
  (−15 пунктов); CodeRankEmbed 0.824 → 0.680. При этом R@10 почти не изменился (0.967→0.944).
- **Диагноз:** картина «R@1 рухнул, R@10 держится» = правильный ответ остаётся в пуле, но его
  вытесняют с верхних позиций. Причины складываются:
  1. RRF с равными весами даёт слабому BM25 такой же «голос», как сильному dense. Здесь dense
     почти у потолка (R@10 0.97), а BM25 слаб → его ложные срабатывания портят топ.
  2. Наш **честный eval враждебен к BM25**: докстринги вырезаны из документов, а запрос — это
     докстринг (английский текст). BM25 сопоставляет слова запроса с кодом БЕЗ докстрингов,
     т.е. только с токенами идентификаторов — лексического пересечения мало. В «прод»-режиме
     (докстринги на месте) BM25 был бы сильнее.
  - Это согласуется с литературой: RRF с равными весами предполагает сопоставимые по силе
    ретриверы. CodeRAG видел «hybrid > dense», потому что у них dense был слабее (bge-small
    ~45 CoIR) — было куда расти.
- **Правка (текущий прогон):** понизили `hybrid_bm25_weight` до 0.3. Ожидание — гибрид как
  минимум догонит dense, а по Recall@50/100 может добавить пропущенные ответы в пул. Проверяем.

### Починка/статус экзотических моделей (по логам последнего прогона)

- **Qodo-1.5B** — `Qwen2Config has no attribute rope_theta`: их кастомный код лезет за
  `rope_theta`. Фикс — подставили `rope_theta: 1000000` через `st_config_kwargs`.
- **jina-v2-base-code** — `cannot import find_pruneable_heads_and_indices from transformers`:
  их код под старую версию transformers, символ удалён. Конфигом не лечится (нужен старый
  transformers, конфликт с Qwen3). Отложено как несовместимость окружения.
- **SFR-Code-400M** — CUDA device-side assert в их rotary; флаги unpad не помогли. Та же
  проблема кастомного кода под другую версию. Отложено.
- Итого рабочих ретриверов для выбора: Qwen3-0.6B, CodeRankEmbed, gte-modernbert, MiniLM,
  t5-base, (Qodo — если фикс сработал) + 2 гибрида.

### Прочие находки прогона

- **t5-base БЕЗ дообучения — R@1 0.43, хуже даже MiniLM (0.49).** Прямо объясняет, зачем
  sturdy-dev его дообучал: сырая general-модель на коде слаба. Иллюстрация ценности адаптации.
- **php = 0 у всех моделей** — баг пайплайна для php (не модели; одинаковый ноль у сильных и
  слабых). Тянет overall вниз одинаково, относительное сравнение корректно. Отдельная починка.

## Сравнение би-энкодеров — предварительный прогон (12.07.2026, Colab T4)

Протокол: python + go, 1000 записей корпуса (по 500 на язык, **общий индекс**),
по 200 запросов на язык, **без реранкера** (чистый retrieval), честный режим
(докстринги вырезаны), chunking 1500/150.
Ветка `feat/embedder-comparison`, ноутбук `colab_embedders.ipynb`, конфиги в `configs/`.

**Python (200 запросов):**

| Embedder | Размер | Recall@1 | Recall@5 | Recall@10 | MRR | NDCG@10 |
|---|---|---|---|---|---|---|
| **CodeRankEmbed** | **137M** | **0.8050** | 0.9200 | 0.9350 | **0.8498** | **0.8706** |
| Qwen3-Embedding-0.6B (baseline) | 600M | 0.7850 | 0.9200 | 0.9350 | 0.8476 | 0.8695 |
| all-MiniLM-L6-v2 (контроль) | 23M | 0.5300 | 0.8100 | 0.8600 | 0.6471 | 0.6991 |

**Go (200 запросов): у ВСЕХ моделей ~0.0000 по всем метрикам** (у MiniLM ровно 1
попадание из 200). Это не свойство моделей — это дефект пайплайна/протокола для go,
разбирается ниже.

**Упавшие модели (заменены/починены в конфигах):**
- Qodo-Embed-1-1.5B — их remote code (`modeling_qwen.py`) несовместим со свежим
  transformers (`Qwen2Config has no attribute rope_theta`). Фикс: `embedder_trust_remote_code: false`
  (архитектура — обычный Qwen2, transformers умеет нативно).
- CodeSage-base-v2 — remote code импортирует `Conv1D` из `transformers.modeling_utils`,
  удалённый много версий назад; чинится только их стороной. Заменён на **gte-modernbert-base** (149M, нативная поддержка).
- SFR-Embedding-Code-400M — CUDA device-side assert в их rotary (Alibaba new_impl,
  `rope_cos[position_ids]`). Заменён на **jina-embeddings-v2-base-code** (161M, код-специфичный).

**Выводы (по python):**
1. **CodeRankEmbed (137M) ≥ Qwen3-0.6B (600M)**: R@1 0.805 vs 0.785 при размере в 4.4
   раза меньше. Кандидат на новый дефолт (особенно для CPU/демо).
2. Контроль сработал: MiniLM заметно хуже — бенчмарк различает модели.
3. R@5/R@10 у топ-двух совпали до 4-го знака (0.9200/0.9350 = 184 и 187 из 200) —
   правдоподобная случайность, но перепроверить на большем корпусе.
4. R@10 ≈ 0.93 на 200 запросах — питон близок к насыщению; для различения моделей
   на большом корпусе (Данил, 100к+) разрывы станут виднее.

### Загадка go=0 (в работе)

Гипотезы: (а) контаминация общего индекса — go-запросы «забиваются» питоновскими
документами, т.к. питон лексически ближе к английскому запросу; (б) баг
парсинга/оценки go (tree-sitter может не разбирать голую функцию без `package`).
Проверка: go-only прогон на MiniLM локально. Результат — здесь после прогона.

## Заметки

- 2026-07-12: до фикса реранкера промпт длиной >512 токенов терял тег
  `<|im_start|>assistant` — модель предсказывала продолжение кода, а не yes/no,
  скоры были мусорными при весе 0.75. Смоук-тест после фикса: релевантная
  функция P(yes)=0.99, нерелевантная 0.00, длинная (4k токенов) с релевантным
  началом 0.13 — порядок корректный.
- Машина для локальных прогонов CPU-only → прогоны уменьшенные (200 записей).
  Полноразмерные прогоны надо гонять на GPU (у Данила / Colab).
- Дефолт `embedding_dtype: float16` на CPU медленный и численно плохой —
  для CPU-прогонов ставить `float32`.
- На C: мало места (кеш HF съел диск, датасет не влез) — перед прогонами
  ставить `HF_DATASETS_CACHE=D:\hf-datasets-cache`, arrow-файлы CodeSearchNet
  занимают несколько ГБ.
